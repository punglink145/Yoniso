# Why-Chain Quality Reference — Yoniso v3.1.0

The 4 quality gates applied to every why-chain layer. A layer failing any gate
is rejected and must be rewritten.

---

## Gate 1: SPECIFICITY — Names a Concrete Identifier

**Check:** Grep for at least one of: function name, file:line, variable name,
config key, error code, API endpoint, condition, type name.

### Pass Examples

- `flush() in BufferPool.release() at buffer_pool.py:142 skips the fsync path when use_async=false`
- `UserService.authenticate() at auth.py:89 returns None for expired tokens instead of raising TokenExpired`
- `rate_limit config key missing from production.yaml — defaults to 0 (unlimited)`
- `react-dom.development.js:22801 — setState on unmounted component in SearchResults`

### Fail Examples

- "there was a sync issue" — no identifier
- "the function didn't work" — no identifier
- "something went wrong in the auth flow" — no identifier
- "the database had a problem" — no identifier

---

## Gate 2: CAUSALITY — Uses Causal Language

**Check:** Must contain at least one of: "because", "which caused", "allowing",
"without", "due to", "as a result of", "leading to", "therefore".

### Pass Examples

- "because `validate_input()` returns None for empty strings, which caused downstream to dereference without checking"
- "the migration ran without a transaction, allowing partial schema under concurrent writes"
- "due to the missing `await` on the async call, the promise was unhandled, leading to silent data loss"

### Fail Examples

- "the input was empty" — statement of fact, not cause
- "it broke after the refactor" — correlation, not causation
- "the user got an error" — observation, not causation
- "probably a race condition" — speculation, not causal

---

## Gate 3: NOVELTY — Not a Restatement of the Layer Above

**Check:** L(N) must introduce new information not present in L(N-1). If L(N)
can be generated by paraphrasing L(N-1), it fails.

### Example of PASS

- **L1:** "Null pointer dereference at `user.email` because `get_user()` returned None"
- **L2:** "`get_user()` can return None because the API contract doesn't guarantee a non-null return, and no caller-side guard exists at any of the 12 call sites — the function was designed for internal use but got exposed as public API in v2.1"

L2 introduces: API contract gap, 12 call sites, design-intent vs usage mismatch.
None of this was in L1.

### Example of FAIL

- **L1:** "Null pointer dereference at `user.email` because `get_user()` returned None"
- **L2:** "user.email was accessed on a null user object" — restates L1

L2 is a paraphrase. It introduces no new information.

---

## Gate 4: ACTIONABILITY — Identifies Something Changeable

**Check:** Names at least one of: code to add/change/remove, config to set, test
to write, hook/lint rule to add, process to change, monitoring to add.

### Pass Examples

- "Add null guard at `auth.py:42` + add `strictNullChecks` to tsconfig"
- "Add `@rate_limit(5/min)` decorator on the login endpoint"
- "Add test category: `test_return_null` for all public API functions"
- "Add pre-commit lint rule: no `await` inside `try` without `catch`"

### Fail Examples

- "users might try to brute force passwords" — not changeable
- "the system is complex" — not changeable
- "we should be more careful" — not a concrete change
- "the code quality is low" — not actionable

## Shallow Output Criteria

After completing the why-chain, produce a **shallow output** for the user.
The shallow output must be decision-bearing, not a summary.

### Decision-Bearing Test

If you can delete the shallow output without changing what the reader would do
next → it fails.

### Examples

**PASS:** "Fix: add null guard at `auth.py:42` + add `strictNullChecks` to tsconfig. Test: add `test_get_user_returns_null` to auth_test.py."

**PASS:** "Revert the migration `V042` — it drops `email` column without backup. Apply again after adding `--backup-table email_backup_20260608`."

**FAIL:** "In summary, we analyzed the null pointer issue and found that adding a guard would prevent it in the future. The analysis above shows that the root cause is the missing null check." — This is a summary, not a decision. Contains no new action information.

### Length Check (Symptom Only)

If shallow output > 30% of deep reasoning length → investigate. But passing
length does NOT mean passing quality. A short summary is still a summary.

### Action-First Format

Lead with the action, not the explanation:
- Good: "Fix: add X at Y. Because: Z."
- Bad: "After careful analysis of Z, we determined that X should be added at Y."
